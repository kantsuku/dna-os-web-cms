# 05. Web公開基盤（薄い独自CMS）設計

> ACMSの責務範囲・最低限機能・不要機能を明確に定義する

---

## 1. 責務の三者分離

### DNA-OS側の責務
- 医院人格・原本データの保持
- 記事原稿の生成（clinic-page-generator経由）
- マークアップ済みHTMLの提供
- 変更イベントの通知

### WebチャネルAI（AI幕僚長 + チャネルAI）の責務
- 影響範囲の分析
- 修正案の自動生成
- メタ情報の生成・最適化
- 品質チェック（誤字脱字、リンク切れ、禁止表現）
- コンプライアンスチェック

### CMS（ACMS）の責務
- **受け取り**: DNA-OS/generatorからのコンテンツ取り込み
- **世代管理**: ページ世代の記録・比較・ロールバック
- **微調整**: 人間による微細編集（差分記録付き）
- **デザイン結合**: コンテンツHTML + デザイントークン + コンポーネント → 最終HTML
- **承認**: 人間が公開判断を下すUI
- **ビルド**: Blade/テンプレート → 静的HTML生成
- **デプロイ**: FTPでXServerに配信
- **上書き制御**: セクション単位のロック・保護
- **例外コンテンツ管理**: 症例・高リスクコンテンツの専用フロー

---

## 2. CMSの設計原則

### 原則1: 「薄さ」の定義
CMSは **コンテンツを生成しない**。受け取ったコンテンツを、デザインと結合し、公開を統制するだけ。

### 原則2: AIが扱いやすい粒度
ページ全体ではなく **セクション単位** でコンテンツを管理する。これにより：
- AIが「この見出し以下を差し替えて」と指示できる
- 人間が「この段落だけロックしたい」と制御できる
- 再生成時に保護されたセクションをスキップできる

### 原則3: 世代管理で安全性を担保
すべての変更は新しい世代として記録。いつでもロールバック可能。

### 原則4: デザインとコンテンツの完全分離
コンテンツHTMLはセマンティックなコンポーネントクラスのみ使用。見た目はデザイントークン + CSSで制御。

## 3. ページ構造モデル

```
Site
  └── Page
        ├── meta (title, description, og:image, ...)
        ├── page_type (top / lower / blog / news / case)
        ├── template_key ("treatment_detail" / "staff" / "custom" ...)
        │
        └── PageGeneration (世代)
              ├── generation: 1, 2, 3...
              ├── source: ai_generated / manual / imported
              ├── sections: [
              │     {
              │       section_id: "sec_01",
              │       heading: "虫歯治療とは",
              │       content_html: "<div class='com-section'>...</div>",
              │       lock_status: unlocked / human_locked / system_locked,
              │       last_modified_by: ai / human:{user_id}
              │     },
              │     ...
              │   ]
              ├── human_patches: [{section_id, patch, reason, user_id, at}]
              ├── final_html: (sections結合 + human_patches適用後)
              └── status: draft → ready → approved → published → superseded
```

### v2からの変更点
v2ではページ全体を1つのcontent_htmlとして管理していたが、v3では **セクション配列** として管理する。これにより：
- セクション単位のロック制御が可能
- 部分再生成が可能
- 差分表示がセクション単位で見やすい

### セクション分割ルール
- `<section class="com-section ...">` タグをセクション境界とする
- clinic-page-generatorの出力がこの構造に準拠済み
- 取り込み時に自動パースしてsections配列に分解

## 4. 最低限必要な機能

### 4-1. サイト管理
| 機能 | 説明 |
|------|------|
| サイトCRUD | 医院単位のサイト登録・編集・削除 |
| FTP設定 | XServerの接続情報管理（暗号化保存） |
| FTP接続テスト | ワンクリックで接続確認 |
| clinic_id紐づけ | DNA-OSのクリニックIDとの対応 |

### 4-2. ページ管理
| 機能 | 説明 |
|------|------|
| ページCRUD | ページの作成・編集・削除 |
| ページ一覧 | ステータス・世代・最終更新でフィルタ |
| メタ情報管理 | title, description, og:image等 |
| ページタイプ設定 | top / lower / blog / news / case |
| テンプレート選択 | ページタイプに応じたテンプレート |

### 4-3. セクション管理（v3新規）
| 機能 | 説明 |
|------|------|

| セクション個別表示 | HTMLプレビュー |
| ロック制御 | unlocked / human_locked / system_locked |
| セクション単位の差分表示 | 旧世代との比較 |

### 4-4. コンテンツ取り込み

| Google Docs URL取り込み | URLからHTMLを取得 |
| マークアップHTML取り込み | _markup.txt からの直接取り込み |
| セクション自動パース | com-section単位に分解 |

### 4-6. 差分表示
| 機能 | 説明 |
|------|------|
| 世代間比較 | 任意の2世代間の差分表示 |
| セクション単位差分 | セクション単位でのハイライト |
| DNA-OS原本との比較 | 原本 vs CMS上のコンテンツ |

### 4-7. 承認
| 機能 | 説明 |
|------|------|
| 承認待ち一覧 | 全サイトの承認待ちを集約 |
| 差分プレビュー付き承認 | 変更内容を確認してから承認 |
| 一括承認 | 低リスクタスクの一括処理 |
| 却下 + コメント | 却下理由の記録 |

### 4-8. 公開（ビルド + デプロイ）
| 機能 | 説明 |
|------|------|
| サイトビルド | 承認済み世代 → 静的HTML生成 |
| プレビュー | ビルド結果のプレビュー（ローカル） |
| FTPデプロイ | XServerへのアップロード |
| デプロイ確認 | デプロイ結果の表示 |

### 4-9. バージョン管理 + ロールバック
| 機能 | 説明 |
|------|------|
| デプロイ履歴 | 全デプロイ記録の一覧 |
| 世代スナップショット | デプロイ時の全ページ世代を記録 |
| ロールバック | 過去のデプロイ状態に戻す |
| ロールバック確認 | 影響範囲を表示してから実行 |

### 4-10. 上書き制御
| 機能 | 説明 |
|------|------|
| セクションロック | 人間が特定セクションをロック |
| ロック警告 | AI再生成時にロックセクションを警告 |
| ロック解除 | 明示的な操作でのみ解除 |
| 強制上書き | admin権限でのみ可能（警告 + 記録） |

### 4-11. 例外コンテンツ管理
| 機能 | 説明 |
|------|------|
| 症例CRUD | 専用入力フォーム |
| コンプライアンスチェック | 自動チェック + 結果表示 |
| 二段階承認 | 一次承認 + 最終承認 |
| 表示制御 | 公開範囲・公開期限の設定 |
| 高リスクフラグ | 医療広告GL関連コンテンツの明示 |

## 5. 不要 / 後回し機能

| 機能 | 理由 |
|------|------|
| 万能ブログCMS | ブログはアシスト執筆。高機能エディタは不要 |
| 高機能WYSIWYG | コンテンツはAIが生成。人間はHTML微調整のみ |
| ノーコードページビルダー | テンプレート+セクション差し込みで十分 |
| AI幕僚長のフル自律実装 | MVPは手動トリガー+ルールベース |
| 画像アップロード/管理 | 初期はパス指定のみ。画像管理は後回し |
| コメント機能 | 不要 |
| 多言語対応 | 不要 |
| A/Bテスト | 後回し |
| リアルタイム共同編集 | 不要 |

## 6. ビルドエンジンの設計

### ビルドフロー

```
1. 対象サイトの全ページを取得
2. 各ページの current_generation を取得
3. final_html を取得（sections結合 + human_patches適用済み）
4. テンプレートに流し込み
   - layout.blade.php (共通レイアウト)
   - {template_key}.blade.php (ページタイプ別)
5. デザイントークンからCSS生成
6. 静的HTMLとして出力
7. 出力ディレクトリに保存
```

### テンプレート階層

```
layouts/
  └── site.blade.php          ← 共通レイアウト（header/footer/nav）
templates/
  ├── top.blade.php           ← TOPページ（自由レイアウト）
  ├── treatment_detail.blade.php  ← 診療詳細（構造化）
  ├── staff.blade.php         ← スタッフ紹介
  ├── facility.blade.php      ← 設備紹介
  ├── recruitment.blade.php   ← 採用
  ├── faq.blade.php           ← FAQ
  ├── blog_post.blade.php     ← ブログ記事
  ├── news_post.blade.php     ← お知らせ
  ├── case_detail.blade.php   ← 症例詳細
  └── generic.blade.php       ← 汎用
```

### 出力ディレクトリ構造

```
build/{site_id}/
  ├── index.html              ← TOPページ
  ├── assets/
  │   ├── css/
  │   │   └── style.css       ← デザイントークンから生成
  │   ├── js/
  │   └── img/
  ├── mushiba/
  │   └── index.html          ← 診療ページ
  ├── blog/
  │   ├── index.html          ← ブログ一覧
  │   └── {slug}/
  │       └── index.html      ← ブログ記事
  ├── case/
  │   ├── index.html          ← 症例一覧
  │   └── {slug}/
  │       └── index.html      ← 症例詳細
  └── ...
```

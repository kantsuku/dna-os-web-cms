# 07. コンテンツ分類と上書き制御設計

> 3種のコンテンツ分類と、再生成・微調整衝突を防ぐ上書き制御ルール

---

## 1. コンテンツ3分類

| 分類 | 対象 | 特性 |
|------|------|------|
| **標準管理コンテンツ** | 診療詳細, FAQ, 医院紹介, 設備紹介, 採用下層 | AI生成・AI更新が主。人間は微調整のみ |
| **アシスト執筆コンテンツ** | ブログ, コラム, お知らせ, イベント | 人間主導。AIはアシスト（校正・提案） |
| **例外管理コンテンツ** | 症例, 医療広告GL配慮文言, 効果訴求表現 | 高リスク。専用承認フロー必須 |

---

## 2. 標準管理コンテンツ

### 対象ページタイプ
- `treatment_detail`（診療詳細）
- `faq`（FAQ）
- `about`（医院紹介）
- `facility`（設備紹介）
- `recruitment`（採用）
- `staff`（スタッフ紹介）

### 属性定義

| 項目 | 定義 |
|------|------|
| **原本** | DNA-OS（マスターデータ） + clinic-page-generator（生成物） |
| **CMSで持つべき情報** | 世代管理されたセクションHTML、メタ情報、ロック状態 |
| **再生成可否** | ○ 条件付き（ロックされたセクションを除く） |
| **承認レベル** | DNA-OS更新起因: risk_level依存 / その他: 管理者承認 |
| **表示制御** | 公開 / 非公開 の2値 |
| **修正可能主体** | AI（再生成・部分修正）、管理者（微調整） |
| **上書きルール** | ロックされていないセクションは再生成で上書き可 |
| **保護ルール** | human_locked セクションはAI上書き不可 |

### 再生成時のフロー

```
再生成要求
  │
  ├── セクション単位でロック状態をチェック
  │
  ├── unlocked セクション → 再生成で上書き
  │     新しい世代として保存
  │
  ├── human_locked セクション → スキップ
  │     旧世代のセクションをそのまま引き継ぎ
  │     ログに「スキップ理由」を記録
  │
  └── system_locked セクション → スキップ
        例外コンテンツとして保護
```

## 3. アシスト執筆コンテンツ

### 対象ページタイプ
- `blog_post`（ブログ）
- `news_post`（お知らせ）

### 属性定義

| 項目 | 定義 |
|------|------|
| **原本** | CMS上の人間入力 or AI下書き（DNA-OSには原本なし） |
| **CMSで持つべき情報** | 本文HTML、メタ情報、公開ステータス、著者情報 |
| **再生成可否** | × 自動再生成不可（人間主導のため） |
| **承認レベル** | 管理者承認必須（クライアント執筆の場合） |
| **表示制御** | 下書き / 公開申請中 / 公開 / 非公開 |
| **修正可能主体** | クライアント（執筆）、AI（校正提案）、管理者（承認・編集） |
| **上書きルール** | AI自動上書き禁止。人間の編集が常に優先 |
| **保護ルール** | 全体がhuman_locked相当 |

### AIアシストの範囲

| アシスト機能 | 説明 | 自動/提案 |
|------------|------|----------|
| 誤字脱字チェック | テキスト校正 | 提案（人間が採用/却下） |
| 禁止表現チェック | 医療広告GL違反表現の検出 | 自動検出 + 警告 |
| DNA整合性チェック | DNA-OS人格との矛盾がないか | 提案 |
| SEO提案 | キーワード提案、文字数チェック | 提案 |
| 文章改善提案 | 読みやすさ、トーン統一 | 提案 |

## 4. 例外管理コンテンツ

### 対象
- 症例（`case_detail`）
- 医療広告GLで慎重な文言を含むセクション（任意のページ内）
- 効果訴求が絡む表現

### 属性定義

| 項目 | 定義 |
|------|------|
| **原本** | CMS上の人間入力（DNA-OSには原本なし） |
| **CMSで持つべき情報** | 構造化データ、コンプライアンスチェック結果、承認記録 |
| **再生成可否** | × 完全禁止。手動修正のみ |
| **承認レベル** | 二段階承認必須（一次: 管理者、最終: 院長/法務） |
| **表示制御** | 非公開 / 限定公開 / 全体公開 / 公開期限付き |
| **修正可能主体** | 管理者のみ（クライアント直接編集不可） |
| **上書きルール** | AI上書き完全禁止（system_locked） |
| **保護ルール** | 修正時は承認フローを再度通す |

### 症例コンテンツの構造化データ

```
ExceptionContent (type: case) {
  title:             "インプラント治療（60代男性）"
  content_html:      "..." (本文HTML)
  structured_data: {
    chief_complaint:  "右下奥歯の欠損"
    treatment:        "インプラント埋入（1本）"
    duration:         "約6ヶ月"
    cost:             "385,000円（税込）"
    risks:            "感染、神経損傷、インプラント周囲炎のリスク"
    age_gender:       "60代男性"
    images: [
      { type: "before", path: "/case/001/before.webp", alt: "治療前" },
      { type: "after",  path: "/case/001/after.webp",  alt: "治療後" }
    ]
  }
  compliance_notes:   "厚労省GL準拠チェック済み。リスク副作用の記載あり"
  status:             draft → first_review → final_review → approved → published
  first_approved_by:  {user_id}
  first_approved_at:  timestamp
  final_approved_by:  {user_id}
  final_approved_at:  timestamp
}
```

## 5. 上書き制御の設計

### 5-1. セクション単位のロック状態

| ロック状態 | 説明 | AI上書き | 人間編集 | 解除方法 |
|-----------|------|---------|---------|---------|
| `unlocked` | 通常状態 | ○ 可 | ○ 可 | — |
| `human_locked` | 人間が微調整した後にロック | × 不可 | ○ 可 | 人間が明示的に解除 |
| `system_locked` | 例外コンテンツとして保護 | × 不可 | △ 承認フロー経由のみ | admin権限 + 理由記録 |

### 5-2. ロック制御のフロー

```
セクション編集
  │
  │ 人間が微調整を保存
  ▼
「このセクションをロックしますか？」
  │
  ├── Yes → human_locked に変更
  │         以降のAI再生成時にスキップ
  │
  └── No → unlocked のまま
            次回AI再生成時に上書きされる可能性あり
            （ただし世代管理で元に戻せる）
```

### 5-3. 再生成と微調整の衝突防止

**シナリオ**: 人間が虫歯治療ページのsec_03を微調整済み。その後DNA-OSの診療方針が更新され、再生成が必要になった。

```
再生成タスク発生
  │
  │ セクション状態チェック
  ▼
sec_01: unlocked → 再生成OK
sec_02: unlocked → 再生成OK
sec_03: human_locked → スキップ
  │
  │ 新世代を作成
  ▼
新世代 = {
  sec_01: 新しいHTML（再生成）
  sec_02: 新しいHTML（再生成）
  sec_03: 旧世代のHTML（引き継ぎ）+ human_locked維持
}
  │
  │ 差分表示
  ▼
管理者に通知:
  「sec_03はロックされているため再生成をスキップしました。
   DNA-OSの変更がsec_03に影響する可能性があります。
   確認してください。」
```

### 5-4. 強制上書き

admin権限者のみ使用可能。

```
管理者: 「sec_03のロックを解除して再生成する」
  │
  │ 確認ダイアログ
  │ 「ロックを解除すると、人間の修正が上書きされます。
  │   修正内容は世代履歴に残ります。続行しますか？」
  ▼
解除 → unlocked に変更
  → 再生成実行
  → 新世代保存
  → ログに「強制上書き: {user_id}, {reason}」記録
```

## 6. コンテンツ分類の判定ルール

### ページタイプによる自動分類

| page_type | 分類 | 理由 |
|-----------|------|------|
| top | 標準管理 | slot部分のみCMS管理 |
| treatment_detail | 標準管理 | AI生成が主 |
| staff | 標準管理 | DNA-OSデータが原本 |
| facility | 標準管理 | DNA-OSデータが原本 |
| about | 標準管理 | DNA-OSデータが原本 |
| recruitment | 標準管理 | DNA-OSデータが原本 |
| faq | 標準管理 | AI生成が主 |
| blog_post | アシスト執筆 | 人間が書く |
| news_post | アシスト執筆 | 人間が書く |
| case_detail | 例外管理 | 医療広告GL |
| contact | — | CMS管理対象外 |

### セクション単位の例外フラグ

標準管理コンテンツのページ内でも、特定のセクションに例外フラグを付けられる。

例: 診療詳細ページ内に「効果訴求を含むセクション」がある場合
→ そのセクションだけ `system_locked` にして、承認フロー必須にする

## 7. まとめ: 分類×操作のマトリクス

| 操作 | 標準管理 | アシスト執筆 | 例外管理 |
|------|---------|------------|---------|
| AI自動生成 | ○ | △（下書きのみ） | × |
| AI再生成 | ○（unlocked部分） | × | × |
| AI修正提案 | ○ | ○（提案のみ） | × |
| 人間微調整 | ○ | ○ | △（承認フロー経由） |
| セクションロック | ○ | —（全体locked） | —（全体locked） |
| 自動デプロイ | △（低リスクのみ） | × | × |
| 承認レベル | 可変 | 管理者必須 | 二段階必須 |
| ロールバック | ○ | ○ | ○（承認記録も復元） |

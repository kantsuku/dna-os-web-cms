# 04. Webチャネル接続導線設計（全パターン）

> DNA-OSからWeb公開までの全導線を、7パターンに分けて完全定義する

---

## 導線一覧

| # | 導線名 | トリガー | 頻度 |
|---|--------|---------|------|
| P1 | DNA-OS更新 → Web反映 | DNA-OS Proposal反映 | 週1〜月数回 |
| P2 | 改善レポート → Web改善タスク | AI幕僚長レポート生成 | 月1〜2回 |
| P3 | フリー入力修正依頼 → Web修正タスク | 人間の自由入力 | 随時 |
| P4 | 新規ページ生成依頼 → Web生成タスク | 人間 or AI幕僚長 | 月0〜数回 |
| P5 | 構造化データ変更 → サイト全体差分反映 | DNA-OS一括変更 | 年数回 |
| P6 | ブログ執筆 → チェック → 承認 → 公開 | クライアント / AI | 週0〜2回 |
| P7 | 症例作成 → 専用承認 → 表示制御 → 公開 | クライアント / 人間 | 月0〜数回 |

## P1. DNA-OS更新 → Web反映

### フロー図

```
DNA-OS Proposal反映
  │
  │ ① 変更イベント発火
  ▼
AI幕僚長: 影響分析
  │ destination_sheet / field から影響ページを特定
  │
  │ ② 戦略タスク生成
  ▼
L3 ダッシュボード: 差分表示 + 承認UI
  │ DNA-OS原本の変更内容を視覚的に表示
  │ 影響ページの一覧を提示
  │
  │ ③ 承認（risk_level=high は必須、low は自動承認可）
  ▼
AI幕僚長: チャネルタスク分解
  │
  │ ④ Webチャネルタスク配信
  ▼
L4 Web: タスク受信
  │
  ├── ④a 記事再生成が必要な場合
  │     → clinic-page-generator 起動
  │     → 新世代HTML取得
  │     → ページ世代として保存
  │
  ├── ④b 部分更新で済む場合
  │     → 該当セクション特定
  │     → AI修正案生成
  │     → 差分プレビュー
  │
  │ ⑤ 微調整（人間、任意）
  ▼
L4 Web: 承認画面
  │ 変更前後の差分表示
  │ 上書き保護セクションの警告
  │
  │ ⑥ 承認
  ▼
L4 Web: ビルド → デプロイ
  │
  │ ⑦ 結果返却
  ▼
L3 ダッシュボード: タスク完了表示
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | DNA-OS の `reflectProposal()` 実行後、90_Update_Logに書き込まれた時点 |
| **入力データ** | proposal_id, destination_sheet, destination_field, proposed_value, before_value |
| **解釈主体** | AI幕僚長（影響マッピングテーブルに基づく） |
| **Web側の処理** | 影響ページ特定 → 再生成 or 部分修正 → 差分プレビュー |
| **承認ポイント** | ① 戦略タスク承認（L3）、② 修正案承認（L4） |
| **公開ポイント** | L4で承認後、ビルド→FTPデプロイ |
| **ログ/証跡** | strategic_task_id, channel_task_id, diff_html, approved_by, deployed_at |
| **結果返却** | タスクステータス更新 + デプロイ記録 |

### 変更イベントの検知方法（MVP）

DNA-OS（GAS）側に新しいAPIエンドポイントを追加：

```
GET /exec?action=getRecentReflections&since=YYYY-MM-DDTHH:mm:ss&clinic_id=XXX
→ [{proposal_id, destination_sheet, destination_field, proposed_value, reflected_at}]
```

ACMS側でcron（5分ごと）でポーリング。将来的にはwebhookに移行。

## P2. 改善レポート → Web改善タスク

### フロー図

```
[L2 データ or 手動入力]
  │
  │ ① パフォーマンスデータ提供
  ▼
AI幕僚長: 改善レポート生成
  │ SEO評価、CTR分析、コンテンツ品質チェック
  │
  │ ② 改善提案をタスク化
  ▼
L3 ダッシュボード: レポート表示
  │ 改善提案の一覧
  │ 優先度・影響範囲の可視化
  │
  │ ③ 人間が採用する提案を選択・承認
  ▼
AI幕僚長: チャネルタスク分解
  │
  │ ④ Webチャネルタスク配信
  ▼
L4 Web: 実行（P1の④以降と同じフロー）
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | 人間がダッシュボードで「レポート生成」を実行 / スケジューラ（月次） |
| **入力データ** | KPIデータ（GA4, GSC）、サイト内部品質データ |
| **解釈主体** | AI幕僚長（Claude/Gemini API） |
| **Web側の処理** | メタ情報改善、コンテンツ改善、内部リンク修正等 |
| **承認ポイント** | ① レポート内の提案承認（L3）、② 修正案承認（L4） |
| **公開ポイント** | L4で承認後、ビルド→デプロイ |
| **ログ/証跡** | improvement_report_id, strategic_task_id, 変更内容 |
| **結果返却** | 改善前後のKPI比較（次回レポートで評価） |

### MVP方針
MVPではAI幕僚長のレポート自動生成は後回し。人間が手動でレポートを作成し、それをタスクに変換する機能のみ実装。

## P3. フリー入力修正依頼 → Web修正タスク

### フロー図

```
人間: フリーテキスト入力
  │ 「トップページの写真を変えたい」
  │ 「虫歯のページの3つ目の見出しを修正して」
  │
  │ ① AI解釈
  ▼
AI幕僚長: 意図解析
  │ 対象チャネル: Web
  │ 対象ページ: TOPページ
  │ 対象セクション: ヒーロー画像
  │ アクション: 画像差し替え
  │
  │ ② 解釈結果を表示、人間が確認
  ▼
L3 ダッシュボード: 解釈確認画面
  │ 「以下の理解で合っていますか？」
  │ [修正] [承認] [キャンセル]
  │
  │ ③ 承認
  ▼
チャネルタスク生成 → L4 Web実行
  │ (P1の④以降と同じフロー)
  ▼
結果返却
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | L3ダッシュボードのフリー入力フォーム送信 |
| **入力データ** | 自然言語テキスト（+ 任意でスクリーンショット） |
| **解釈主体** | AI幕僚長（Claude API: 1回の解釈呼び出し） |
| **Web側の処理** | 対象ページ/セクション特定 → 修正案作成 |
| **承認ポイント** | ① 解釈結果の確認（L3）、② 修正案承認（L4） |
| **公開ポイント** | L4で承認後、ビルド→デプロイ |
| **ログ/証跡** | free_input_id, 元テキスト, AI解釈結果, タスクID |
| **結果返却** | タスクステータス + 変更箇所のリンク |

### AI解釈のプロンプト設計（概要）

```
あなたはWeb修正リクエストの解釈AIです。
ユーザーの自然言語リクエストを解析し、以下を特定してください：

1. 対象ページ（slug or ページタイプ）
2. 対象セクション（セクション番号 or 見出しテキスト）
3. 修正アクション（テキスト変更 / 画像差替 / レイアウト変更 / 削除 / 追加）
4. 修正内容の具体案（可能な場合）
5. DNA-OS原本への影響有無

サイトのページ一覧：
{pages_list}

現在のページ構成：
{page_sections}
```

## P4. 新規ページ生成依頼 → Web生成タスク

### フロー図

```
人間 or AI幕僚長: 新規ページ依頼
  │
  │ ① 前提確認
  ▼
AI幕僚長: DNA-OSデータ確認
  │ 該当診療のデータがDNA-OSに存在するか？
  │ 記事生成に必要な情報が揃っているか？
  │
  ├── 不足あり → DNA-OS更新を先に依頼
  │
  │ ② 記事生成パイプライン起動
  ▼
clinic-page-generator: 記事生成
  │ Gemini(草稿) → Claude(検収/FC/マークアップ)
  │ → Google Docs + _markup.txt 出力
  │
  │ ③ ACMS取り込み
  ▼
L4 Web: 新規ページ作成
  │ マークアップHTMLを取り込み
  │ 新規 PageGeneration 作成
  │ メタ情報をAIで自動生成
  │
  │ ④ 微調整 + 承認
  ▼
L4 Web: プレビュー → 承認 → 公開
  │
  │ ⑤ 内部リンク更新
  ▼
関連ページの内部リンクを自動チェック・更新提案
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | L3ダッシュボードで「新規ページ生成」依頼 |
| **入力データ** | 診療名（internal_key）、ページタイプ、追加指示 |
| **解釈主体** | AI幕僚長（DNA-OSデータ充足確認） |
| **Web側の処理** | 記事生成→取り込み→ページ作成→メタ生成 |
| **承認ポイント** | ① 生成前の確認（L3）、② 生成結果の承認（L4） |
| **公開ポイント** | L4で承認後、ビルド→デプロイ |
| **ログ/証跡** | strategic_task_id, generator実行ログ, page_generation_id |
| **結果返却** | 新規ページURL + 生成レポート |

## P5. 構造化データ変更 → サイト全体差分反映

### フロー図

```
DNA-OS: 大規模更新（理念変更、トーン変更等）
  │
  │ ① 変更イベント（複数Proposal一括反映）
  ▼
AI幕僚長: 影響範囲スキャン
  │ 全ページを対象に影響度を分析
  │ 影響度: high / medium / low に分類
  │
  │ ② 一括戦略タスク生成
  ▼
L3 ダッシュボード: 影響マップ表示
  │ ページ一覧 × 影響度のマトリクス
  │ 高影響ページを優先表示
  │
  │ ③ 一括承認 or 個別承認
  ▼
チャネルタスク: ページごとに分解
  │ 高影響: 再生成 → 全文チェック
  │ 中影響: 部分修正 → 差分チェック
  │ 低影響: 自動修正 → ログのみ
  │
  │ ④ 段階的デプロイ
  ▼
L4 Web: 高影響ページから順に実行・承認・公開
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | DNA-OS上で5件以上のProposalが同一セッションで反映された場合 |
| **入力データ** | 変更Proposal一覧、影響マッピング |
| **解釈主体** | AI幕僚長（全ページスキャン） |
| **Web側の処理** | 影響度別の段階的更新 |
| **承認ポイント** | ① 影響マップの確認（L3）、② 高影響ページの個別承認（L4） |
| **公開ポイント** | 段階的デプロイ（高影響→中→低） |
| **ログ/証跡** | batch_task_id, 影響マップスナップショット, 各ページの変更ログ |
| **結果返却** | 一括更新レポート（成功/失敗/スキップの内訳） |

## P6. ブログ執筆 → チェック → 承認 → 公開

### フロー図

```
クライアント or AI: ブログ下書き作成
  │
  ├── A: クライアントが手書き
  │     → ACMS上のブログエディタで入力
  │     → AIアシスト（文章校正、表現改善、禁止語チェック）
  │
  ├── B: AIが下書き生成
  │     → テーマ/キーワード指定
  │     → AI生成 → 下書きとして保存
  │
  │ ① 品質チェック（自動）
  ▼
L4 Web: 自動チェック
  │ - 禁止表現チェック（医療広告GL）
  │ - 誤字脱字チェック
  │ - DNA-OS人格との整合性チェック
  │ - SEO基本チェック（文字数、見出し構造）
  │
  │ ② チェック結果表示 + 修正提案
  ▼
クライアント or 管理者: 修正
  │
  │ ③ 公開申請
  ▼
管理者: 承認
  │ - 内容確認
  │ - 医療広告GL準拠確認
  │ - ブランドトーン確認
  │
  │ ④ 承認
  ▼
L4 Web: 公開
  │ ビルド → デプロイ
  ▼
完了
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | クライアントがブログ作成画面を開く / AI生成を依頼 |
| **入力データ** | ブログ本文（手書き or AI生成） |
| **解釈主体** | L4 WebチャネルAI（品質チェック） |
| **Web側の処理** | 自動チェック → 修正提案 → 承認フロー |
| **承認ポイント** | 管理者による公開前承認（**必須**） |
| **公開ポイント** | 承認後、ビルド→デプロイ |
| **ログ/証跡** | ブログID, チェック結果, 承認記録, 修正履歴 |
| **結果返却** | 公開URL |

## P7. 症例作成 → 専用承認 → 表示制御 → 公開

### フロー図

```
クライアント or 管理者: 症例データ入力
  │
  │ 専用入力フォーム
  │ - 主訴
  │ - 治療内容
  │ - 治療期間・費用
  │ - リスク・副作用
  │ - 画像（術前・術後）
  │
  │ ① コンプライアンスチェック（自動）
  ▼
L4 Web: 専用チェック
  │ - 医療広告GL準拠（厚労省ガイドライン）
  │   ・治療内容/期間/費用の記載漏れチェック
  │   ・リスク副作用の記載チェック
  │   ・誇大表現チェック
  │ - 画像の適切性確認（メタデータチェック）
  │ - 個人情報の混入チェック
  │
  │ ② チェック結果 + 修正要求
  ▼
管理者: 修正 + 一次承認
  │
  │ ③ 一次承認
  ▼
専門レビュアー（院長 or 法務担当）: 最終承認
  │ - 医学的正確性の確認
  │ - 法的リスクの確認
  │
  │ ④ 最終承認
  ▼
L4 Web: 表示制御設定
  │ - 公開範囲（全体公開 / 限定公開 / 非公開）
  │ - 公開期限（任意）
  │ - カテゴリ・タグ設定
  │
  │ ⑤ 公開
  ▼
L4 Web: ビルド → デプロイ
```

### 詳細定義

| 項目 | 内容 |
|------|------|
| **開始トリガー** | 症例入力フォームの送信 |
| **入力データ** | 症例構造化データ（主訴/治療内容/期間/費用/リスク/画像） |
| **解釈主体** | L4 WebチャネルAI（コンプライアンスチェック） |
| **Web側の処理** | 自動チェック → 修正 → 二段階承認 → 表示制御 → 公開 |
| **承認ポイント** | ① 管理者一次承認、② 専門レビュアー最終承認（**両方必須**） |
| **公開ポイント** | 最終承認後、表示制御設定→ビルド→デプロイ |
| **ログ/証跡** | exception_content_id, チェック結果, 一次承認記録, 最終承認記録 |
| **結果返却** | 公開URL + コンプライアンスチェック結果レポート |

### 上書き制御
- 症例コンテンツは **自動上書き完全禁止**
- DNA-OS更新の影響を受けない
- 修正は必ず人間が行い、承認フローを再度通す

---

## 導線間の相互作用

### P1 → P5 のエスカレーション
P1（単発更新）で影響ページが5つ以上になった場合、自動的にP5（全体差分反映）モードに切り替わる。

### P4 → P1 の連鎖
新規ページ生成（P4）後、既存ページの内部リンクにP1タスクが自動生成される。

### P6/P7 → P2 の連鎖
ブログ・症例が蓄積した後、AI幕僚長がP2（改善レポート）でブログのSEO効果を評価し、追加タスクを生成する。

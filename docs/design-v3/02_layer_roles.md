# 02. 各レイヤーの役割定義

> 全7コンポーネントの責務・入出力・保持情報・接続を定義する

---

## L1. DNA-OS（原本基盤）

### 役割
医院の「人格」と「事実」の唯一の原本。全チャネルの情報源。

### 入力
- 一次情報（ヒアリング、アンケート、院長の言葉）
- クライアント修正（確認画面経由）
- 外部ソース（Google Docs / Spreadsheet URL）

### 出力
- 構造化マスターデータ（11シート）
- 変更イベント（Proposal反映時）
- 記事原稿（clinic-page-generator経由）
- マークアップ済みHTML

### 保持すべき情報
- 医院人格（DNA_Master）
- 診療方針（Treatment_Policy）
- スタッフ情報（Staff_Master）
- 採用方針（Recruitment_Policy）
- クレド（Credo_Master）
- トーン＆マナー（Tone_And_Manner）
- コンテンツ生成ガイド（Content_Gen_Guide）
- 戦略状態（Strategy_State）
- 紹介ネットワーク（Referral_Network）
- 変更提案と監査ログ

### 保持すべきでない情報
- 公開ページの最終HTML
- チャネル固有の表示設定
- デザイントークン・CSS
- 承認フローの状態
- パフォーマンス計測データ

### 他レイヤーとの接続
- → AI幕僚長: 変更イベント通知、マスターデータ提供
- → L4 Web: 記事原稿・マークアップHTML提供
- ← L3: クライアント修正フィードバック（将来）

---

## L2. 分析・データ基盤

### 役割
各チャネルのパフォーマンスデータを収集・正規化し、AI幕僚長に渡す。

### 入力
- Google Analytics 4 データ
- Google Search Console データ
- GMBインサイト
- サイト内部品質データ（リンク切れ、文字数、メタ情報等）

### 出力
- 正規化されたKPIデータ
- チャネル別パフォーマンス指標

### 保持すべき情報
- 計測データの時系列スナップショット
- KPI定義・閾値設定

### 保持すべきでない情報
- 原本コンテンツ
- 戦略判断の結果
- チャネル固有の実装詳細

### 他レイヤーとの接続
- → AI幕僚長: KPIデータ提供
- ← L5: 結果データ受け取り

### MVP方針
**MVP対象外**。初期はAI幕僚長が手動で受け取ったデータを元に動作する。将来的にAPI連携で自動化。

---

## L3. 戦略統制基盤（ダッシュボード）

### 役割
人間がAI幕僚長の提案を確認・承認・指示するUI。戦略の「見える化」と「承認ゲート」。

### 入力
- AI幕僚長が生成した改善レポート
- AI幕僚長が生成した戦略タスク
- 人間のフリー入力（修正依頼・新規指示）
- DNA-OS更新差分の通知

### 出力
- 承認済み戦略タスク
- 人間の修正依頼（→AI幕僚長がタスク化）
- タスクのステータス更新

### 保持すべき情報
- 戦略タスク一覧とステータス
- 改善レポート履歴
- 承認記録
- フリー入力修正依頼ログ

### 保持すべきでない情報
- 原本データそのもの（参照のみ）
- チャネル側の実装詳細
- デザインデータ

### 他レイヤーとの接続
- ← AI幕僚長: レポート・タスク受け取り
- → AI幕僚長: 承認・指示の送信
- → L4: 承認済みチャネルタスクの配信
- ← L4: 実行結果の受け取り

### 画面の位置づけ
L3のダッシュボードとL4のWeb管理画面は **同一Laravelアプリ内** に共存する。ナビゲーション上で「戦略」と「Web管理」を分離。

---

## L4. チャネル実行基盤（Web）

### 役割
チャネルタスクを受け取り、Web固有の解釈・実行・微調整・ビルドを行う。

### 入力
- チャネルタスク（戦略統制基盤から）
- DNA-OS原本データ（記事原稿・マークアップHTML）
- デザイントークン・コンポーネント定義

### 出力
- 修正案（差分プレビュー）
- ビルド成果物（静的HTML）
- 実行結果レポート

### 保持すべき情報
- サイト定義（ドメイン、FTP情報、デザイン設定）
- ページ定義（slug、タイプ、メタ情報）
- ページ世代（content_html、human_patch、final_html）
- コンポーネント定義・デザイントークン
- 例外コンテンツ
- セクション単位のロック状態

### 保持すべきでない情報
- DNA-OSの原本データそのもの（参照・キャッシュのみ）
- 戦略判断の根拠データ
- 他チャネルの情報

### 他レイヤーとの接続
- ← L3: チャネルタスク受け取り
- ← L1: 原本データ参照
- → L5: ビルド成果物のデプロイ指示
- → L3: 実行結果返却

---

## L5. 公開・結果回収層

### 役割
ビルド成果物を本番環境にデプロイし、公開後の結果を回収する。

### 入力
- ビルド成果物（静的HTML + CSS + 画像）
- デプロイ指示

### 出力
- デプロイステータス（成功/失敗）
- 公開URL
- パフォーマンスデータ（将来）

### 保持すべき情報
- デプロイ履歴
- 公開中のファイルスナップショット（ロールバック用）

### 保持すべきでない情報
- 原本データ
- 戦略判断
- CMS管理データ

### 他レイヤーとの接続
- ← L4: ビルド成果物受け取り
- → L2: 結果データの提供（将来）
- → L4: デプロイステータス返却

## 横断: AI幕僚長

### 役割
データを統合し、改善レポートを生成し、戦略タスクに変換し、チャネル実行を監督する**オーケストレータ**。

### 重要な制約
- **DNA-OSの原本を直接書き換えない**（提案は出すが、DNA-OS側のProposal経由）
- **各チャネルの最終公開を独断で行わない**（必ず承認ゲートを通す）
- **チャネル固有の実装判断はしない**（チャネルAIに委譲）

### 入力
- L1: DNA-OS変更イベント、マスターデータ
- L2: KPIデータ、パフォーマンス指標
- L3: 人間の指示、フリー入力修正依頼
- L4: チャネル実行結果

### 出力
- 改善レポート
- 戦略タスク（チャネル横断）
- チャネルタスク（チャネル別に分解）
- 実行監督ログ

### 保持すべき情報
- 改善レポート履歴
- タスク変換ログ
- 監督ログ

### 保持すべきでない情報
- 原本データそのもの
- チャネル固有の実装状態
- デザインデータ

### MVP方針
MVPでは **手動トリガー + 単純なルールベース変換** で最小実装。高度な自律判断は後回し。

具体的には：
1. DNA-OS変更 → Webタスク自動生成（ルールベース）
2. フリー入力 → AI解釈→タスク生成（Claude API 1回呼び出し）
3. 改善レポート → 手動作成（将来自動化）

## レイヤー間の責務境界（FAQ）

### Q: 記事原稿はどこで生成するのか？
**clinic-page-generator（GAS）** が生成する。L4 Webチャネルは生成済みHTMLを「受け取る」だけ。ただし、L4内でのメタ情報生成（title/description）やSEO最適化はL4の責務。

### Q: デザインはどのレイヤーが管理するのか？
**L4 Webチャネル** が管理する。デザイントークン・コンポーネント定義・CSS生成はすべてL4。DNA-OSはデザインに関知しない。

### Q: 承認は誰がどこでするのか？
- **戦略タスクの承認** → L3ダッシュボード上で人間が承認
- **Web修正案の承認** → L4 Web管理画面上で人間が承認
- **例外コンテンツの承認** → L4 Web管理画面上で専用承認フロー

### Q: AI幕僚長はどこに実装するのか？
Laravel内のServiceクラス + AI API呼び出し。L3の画面からトリガーされ、結果をL3のDBに保存する。将来的にはバッチ実行・スケジューラ起動も想定。

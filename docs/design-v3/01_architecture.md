# 01. 全体アーキテクチャ再設計案

> DNA-OS統合作戦基盤 v3 — Webチャネル導線強化版

---

## 1. 設計思想

v1（CMS）→ v2（ACMS）の流れを踏まえ、v3では **DNA-OSを原本基盤として守りつつ、戦略統制→チャネル実行→公開の全導線を一気通貫で設計** する。

### 根本方針
1. **原本はDNA-OSにしかない** — 他レイヤーはすべて「参照・変換・表現」
2. **戦略と実行を分離** — AI幕僚長は「何をすべきか」を決め、チャネルAIは「どうやるか」を決める
3. **人間の介入ポイントは最小限かつ明確** — 承認・微調整・例外判断のみ
4. **チャネルは交換可能** — Web以外のチャネルが来ても同じタスク構造で対応できる

---

## 2. 全体レイヤー構成（5層 + 横断層）

```
┌─────────────────────────────────────────────────────────────┐
│                    横断層: AI幕僚長                          │
│            (統制・タスク変換・監督・レポート生成)               │
├─────────────────────────────────────────────────────────────┤
│ L1  原本基盤              │ DNA-OS (GAS+Spreadsheet)        │
├───────────────────────────┤                                 │
│ L2  分析・データ基盤       │ 計測データ収集・統合              │
├───────────────────────────┤                                 │
│ L3  戦略統制基盤           │ ダッシュボード + 承認UI           │
├───────────────────────────┤                                 │
│ L4  チャネル実行基盤       │ Web / GBP / SNS / LINE          │
├───────────────────────────┤                                 │
│ L5  公開・結果回収層       │ XServer / GMB API / 各API       │
└─────────────────────────────────────────────────────────────┘
```

### なぜ「横断層」か

AI幕僚長をL3に押し込むと「戦略統制＝AI幕僚長」になり、ダッシュボードとの責務が混線する。AI幕僚長はL1〜L5を横断して情報を吸い上げ・指示を出す**オーケストレータ**であり、特定レイヤーに属さない。

---

## 3. 各レイヤーの責務サマリー

| レイヤー | 主な責務 | 入力 | 出力 |
|---------|---------|------|------|
| **L1 DNA-OS** | 医院人格・原本データの保持・更新 | 一次情報（ヒアリング等） | 構造化データ、変更イベント |
| **L2 分析・データ基盤** | 計測データ収集・加工 | GA4, Search Console, GMB等 | KPIデータ、パフォーマンス指標 |
| **L3 戦略統制基盤** | 戦略意図の表示・承認・タスク管理UI | AI幕僚長の出力、人間の指示 | 承認済みタスク、修正依頼 |
| **L4 チャネル実行基盤** | チャネル固有の実行・微調整・ビルド | チャネルタスク、原本データ | 修正案、ビルド成果物 |
| **L5 公開・結果回収** | デプロイ・公開・結果の回収 | ビルド成果物 | 公開ステータス、パフォーマンスデータ |
| **横断: AI幕僚長** | 分析→戦略→タスク変換→監督 | 全レイヤーの情報 | 改善レポート、戦略タスク |

## 4. システム間データフロー

```
[L1 DNA-OS]
    │
    │ ① 変更イベント (webhook / polling)
    ▼
[横断: AI幕僚長]  ←── ② 計測データ ──── [L2 分析・データ基盤]
    │                                         ▲
    │ ③ 改善レポート                            │
    │ ④ 戦略タスク生成                          │
    ▼                                          │
[L3 戦略統制基盤]                               │
    │                                          │
    │ ⑤ 承認済みタスク                          │
    │ ⑥ チャネルタスク分解                       │
    ▼                                          │
[L4 チャネル実行基盤]                            │
    │ (Web / GBP / SNS / LINE)                 │
    │                                          │
    │ ⑦ 修正案作成 → 承認 → ビルド               │
    ▼                                          │
[L5 公開・結果回収]  ─── ⑧ 結果データ ──────────┘
```

## 5. v2からの主な変更点

| 観点 | v2 | v3 |
|------|----|----|
| スコープ | Webチャネルのみ | 全チャネル対応設計（MVP実装はWeb） |
| AI幕僚長 | 未設計 | 横断オーケストレータとして定義 |
| タスク体系 | なし（手動取り込み） | 戦略タスク → チャネルタスクの2段構造 |
| 導線設計 | DNA-OS→取り込み→公開のみ | 7パターンの導線を完全定義 |
| 分析層 | なし | L2として分離定義（MVP後回し） |
| 承認フロー | 公開ボタンのみ | タスク種別×コンテンツ分類で承認レベル可変 |
| 上書き制御 | 世代管理のみ | セクション単位のロック + 再生成衝突防止 |

## 6. 技術スタック（確定）

| レイヤー | 技術 |
|---------|------|
| L1 DNA-OS | Google Apps Script + Spreadsheet |
| L2 分析基盤 | MVP後回し（将来: Laravel + 外部API連携） |

| AI幕僚長 | Gemini / Claude API (GAS or Laravel経由) |
| DB | MySQL 8.0 (XServer互換) |
| フロント | Livewire 4 + Alpine.js + Tailwind CSS |

## 7. マルチテナント設計

```
ACMS (1インスタンス)
  ├── Site: 欅歯科  → XServer A
  ├── Site: 桜歯科  → XServer B
  └── Site: 松歯科  → XServer C

DNA-OS (1インスタンス / GAS)
  ├── Clinic: 欅歯科 → Spreadsheet A
  ├── Clinic: 桜歯科 → Spreadsheet B
  └── Clinic: 松歯科 → Spreadsheet C
```

- ACMSとDNA-OSは `clinic_id` で紐づけ
- サイトごとにFTP認証情報・デザイン設定・ユーザーアクセス権を分離
- AI幕僚長はクリニック横断の情報には触れない（テナント分離厳守）

---

## 8. 非機能要件

| 項目 | 要件 |
|------|------|
| 同時管理サイト数 | 90+ |
| ビルド速度 | 1サイト30秒以内 |
| デプロイ | FTP、リトライ3回、アトミック性確保 |
| 可用性 | ACMS本体はXServerで運用可能 |
| セキュリティ | FTP認証情報は暗号化保存、ロール別アクセス制御 |
| 監査 | 全変更に証跡（タスクID・承認者・タイムスタンプ） |
